## ADDED Requirements

### Requirement: Sparkle dependency declared
The project SHALL declare Sparkle 2 as an SPM package dependency in `project.yml` and link it to the `ClaudeUsage` target.

#### Scenario: XcodeGen resolves Sparkle
- **WHEN** `xcodegen generate` is run
- **THEN** the generated `.xcodeproj` includes a Sparkle SPM package reference pointing to `https://github.com/sparkle-project/Sparkle` at a pinned version tag

---

### Requirement: EdDSA public key embedded in app
The app bundle SHALL contain the EdDSA public key as the `SUPublicEDKey` entry in `Info.plist`, generated by Sparkle's `generate_keys` tool.

#### Scenario: Public key present at runtime
- **WHEN** the app launches
- **THEN** Sparkle can read `SUPublicEDKey` from the bundle's `Info.plist` without error

---

### Requirement: Appcast feed URL declared in app
The app bundle SHALL contain an `SUFeedURL` entry in `Info.plist` pointing to `https://raw.githubusercontent.com/eylonshm/claude-usage-widget/main/appcast.xml`.

#### Scenario: Feed URL resolved
- **WHEN** Sparkle initiates an update check
- **THEN** it fetches from the URL declared in `SUFeedURL`

---

### Requirement: Auto-check enabled by default
The app SHALL automatically check for updates in the background on launch via `SUEnableAutomaticChecks: true` in `Info.plist`.

#### Scenario: Background check on first launch
- **WHEN** the app finishes launching for the first time
- **THEN** Sparkle schedules a background update check without user interaction

#### Scenario: Subsequent launch check interval
- **WHEN** the app finishes launching and the last check was more than 24 hours ago
- **THEN** Sparkle performs a background update check

---

### Requirement: Updater controller initialized in AppDelegate
`AppDelegate` SHALL hold a strong reference to `SPUStandardUpdaterController`, initialized in `applicationDidFinishLaunching`.

#### Scenario: Controller lifetime
- **WHEN** the app is running
- **THEN** the `SPUStandardUpdaterController` instance is alive for the full app lifetime and Sparkle can present update UI at any time

---

### Requirement: Update available prompt shown to user
When a new version is found, Sparkle SHALL present its standard update UI (version notes + Install/Skip buttons).

#### Scenario: New version available
- **WHEN** Sparkle fetches the appcast and finds a version newer than the running app's `CFBundleShortVersionString`
- **THEN** Sparkle displays the standard update prompt with release notes

#### Scenario: No update available (manual check)
- **WHEN** the user clicks "Check for Updates" and no newer version exists
- **THEN** Sparkle displays a "You're up to date" alert

---

### Requirement: Appcast generated and committed in CI
The GitHub Actions release workflow SHALL generate a signed `appcast.xml` using Sparkle's `generate_appcast` tool and commit it to the `main` branch after each release.

#### Scenario: Release tag pushed
- **WHEN** a `v*` tag is pushed and the CI build completes
- **THEN** `appcast.xml` at the repo root is updated with the new release entry (version, URL, EdDSA signature, length)
- **AND** the file is committed to `main` with message `chore: update appcast for v<version>`

#### Scenario: Appcast signature valid
- **WHEN** Sparkle downloads the DMG referenced in `appcast.xml`
- **THEN** the EdDSA signature in the appcast matches the DMG, and Sparkle accepts the update
